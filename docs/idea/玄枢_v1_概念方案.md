# AI 加密合约短线 Trading Harness V1 规格

## 1. 项目目标

本项目旨在构建一个面向 **OKX 永续合约短线交易** 的 **Trading Harness**，而不是一个“让 AI 直接做交易决策”的黑盒系统。

这里的 Harness 指：

- 有明确边界的运行壳
- 有快慢路径分离
- 有结构化状态契约
- 有检查点、回放、恢复和降级能力
- 有评测、晋升与回滚机制
- AI 只是其中的一个治理模块，而不是整个系统本身

V1 的核心目标不是追求极致高频，而是优先完成以下六件事：

1. 跑通稳定的实时行情、信号、下单、风控、回报闭环。
2. 建立明确的 **快路径 / 慢路径** 双路径架构。
3. 建立结构化状态契约、配置快照与执行检查点。
4. 建立回放、仿真、模拟盘到小资金实盘的统一演进路径。
5. 建立可观测、可审计、可恢复、可降级的系统底座。
6. 尽可能复用成熟的开源免费组件，降低研发成本和维护风险。

## 2. Harness 设计原则

### 2.1 模型不是系统，Harness 才是系统

AI 模型只负责系统中的一部分低频治理能力，例如：

- 市场理解
- 策略路由建议
- 风险建议
- 复盘审计
- 记忆检索
- 候选优化方案生成

真正的系统能力来自 Harness：

- 状态管理
- 权限边界
- 配置快照
- 风控硬约束
- 执行状态机
- 检查点与恢复
- 评测与晋升

### 2.2 快路径和慢路径必须物理分离

V1 采用双路径架构：

- **快路径（Fast Path）**：逐笔交易、低延迟、确定性执行
- **慢路径（Slow Path）**：低频治理、AI 分析、审计、记忆、配置发布

快路径绝不能等待慢路径返回结果。

### 2.3 状态优先，不靠长上下文硬撑

系统中所有关键对象必须有结构化状态表示，而不是依赖长 prompt 或聊天式上下文。

核心思想是：

- AI 读取状态对象
- 执行层读取配置快照
- 恢复流程读取检查点
- 评测流程读取审计记录

而不是依赖“把过去发生的一切重新讲一遍”。

### 2.4 先保证可活，再追求高频

V1 首先覆盖 **稳定优先的自动化短线 / 中高频执行** 区间，而不是一开始追求顶级 HFT。系统频率设计必须服从 OKX 官方免费接口限制，并优先考虑：

- 网络波动
- 交易所接口抖动
- AI 调用延迟波动
- 本地机器负载波动
- 状态一致性恢复成本

### 2.5 AI 不进入逐笔交易热路径

AI 不负责：

- 毫秒级逐笔下单决策
- 风控保险丝修改
- 执行状态机变更
- 高频、稳定、低延迟的在线审批
- 直接热更新实盘核心逻辑

V1 明确假设：

**GPT 类模型响应速度存在明显波动，不能作为稳定低延迟组件使用。**

因此 AI 在本系统中被定义为：

**低频治理与研究能力**

而不是：

**实时交易决策引擎**

### 2.6 可观测性、可回放、可恢复是一等公民

每一笔交易必须能回答：

- 为什么做
- 当前生效的是哪一版配置快照
- 是否在授权范围内执行
- 为什么成交
- 为什么盈利或亏损
- 如果系统中断，如何从检查点恢复

## 3. 交易范围与接口边界

本系统 V1 以 **OKX 永续合约** 为核心交易场所，交易标的明确限定为：

- `BTC-USDT-SWAP`
- `ETH-USDT-SWAP`

V1 不扩展到更多 symbol，优先保证系统稳定性、状态一致性与复盘质量。

### 3.1 现实接口边界

基于 OKX 官方文档，V1 设计采用如下现实边界：

- 行情侧重点使用 `bbo-tbt` 与 `books5`
- 单笔下单 / 撤单 / 改单预算按 `60 requests / 2 seconds` 设计
- 批量接口作为后续吞吐提升手段，但 V1 不依赖极限频率运行
- 执行层使用显式动作预算池，主动低于理论限频运行，为异常与重试保留冗余

### 3.2 V1 交易风格

V1 更适合：

- 短线趋势突破
- 短线回归
- 异常行情过滤
- **以 taker 为主的确定性执行**

V1 明确采用：

- **以 taker 为主的执行风格**
- 不以 maker 挂撤为主要获利方式
- 不追求高频做市式库存管理
- 单 symbol 只保留单向仓位管理，不做复杂反手与双向对冲
- **杠杆上限固定为 3x**，V1 不根据 AI 判断动态提升杠杆

V1 不适合：

- 共址级 HFT
- 极高频盘口抢跑
- 对微秒级排队位置极端敏感的策略

## 4. Harness 总体架构

系统由四层组成：

1. **执行与状态热路径层**
2. **快路径规则层**
3. **慢路径治理层**
4. **基础设施层**

### 4.1 双路径关系

- **快路径（热路径）**：行情/账户数据 -> 状态引擎 -> 信号工厂 -> 风控内核 -> 执行引擎 -> 订单回报 -> 状态更新
- **慢路径（治理路径）**：状态摘要/历史记忆/审计结果 -> Expert Layer -> Decision Committee -> 策略配置快照 -> 下发给快路径规则引擎

V1 中 **Expert Layer** 与 **Decision Committee** 不参与每一笔交易的同步审批，而是以低频方式产出“策略配置快照”，供快路径直接读取执行。

## 5. 核心契约对象

Harness 的核心不是“专家人格”，而是 **契约对象**。V1 至少定义 5 类核心对象。

### 5.1 MarketStateSnapshot

用于表达当前市场与系统状态。

建议字段：

- `snapshot_id`
- `generated_at`
- `symbol`
- `mid_price`
- `spread`
- `imbalance`
- `recent_trade_bias`
- `volatility_state`
- `regime`
- `current_position`
- `current_mode`
- `risk_budget_remaining`

### 5.2 StrategyConfigSnapshot

这是快慢路径之间最重要的桥梁对象。

建议字段：

- `version_id`
- `generated_at`
- `effective_from`
- `expires_at`
- `symbol_whitelist`
- `strategy_enable_flags`
- `risk_multiplier`
- `per_symbol_max_position`
- `max_leverage`
- `market_mode`
- `approval_state`
- `source_reason`
- `ttl_sec`

说明：

- 快路径只读取最新一版已生效快照
- 慢路径负责生成、更新、替换快照
- 过期快照不得继续用于新增风险动作

### 5.3 ExpertOpinion

用于规范专家输出，避免自由文本失控。

建议字段：

- `opinion_id`
- `expert_type`
- `generated_at`
- `symbol_scope`
- `decision`
- `confidence`
- `supporting_facts`
- `risk_flags`
- `ttl_sec`

### 5.4 RiskDecision

用于表达风控放行或拒绝结果。

建议字段：

- `decision_id`
- `generated_at`
- `symbol`
- `allow_open`
- `allow_close`
- `max_position`
- `max_order_size`
- `risk_mode`
- `reason_codes`

### 5.5 ExecutionCheckpoint

这是 durable execution 的核心对象。

建议字段：

- `checkpoint_id`
- `created_at`
- `active_snapshot_version`
- `current_mode`
- `positions_snapshot`
- `open_orders_snapshot`
- `budget_state`
- `last_public_stream_marker`
- `last_private_stream_marker`
- `needs_reconcile`

说明：

系统中断或重启时，应先恢复检查点，再与交易所对账，而不是重新猜测当前状态。

## 6. 模块设计

### 6.1 市场数据总线 Market Bus

职责：

- 接入 OKX 公共与私有 WebSocket 数据
- 标准化输出行情、订单簿、成交、持仓、订单状态事件
- 推送给状态层和执行层

输出事件类型：

- `tick_event`
- `orderbook_event`
- `trade_event`
- `position_event`
- `order_update_event`

建议复用：

- **NautilusTrader** 作为事件驱动交易系统底座
- OKX 官方 REST / WebSocket 作为最终数据与交易接入

### 6.2 状态引擎 State Engine

职责：

- 维护当前市场状态
- 计算短周期特征
- 管理仓位、订单、动作预算和市场环境标签
- 产出 `MarketStateSnapshot`

核心状态：

- mid price / spread
- order book imbalance
- 最近主动买卖成交偏置
- 波动率与波动扩张状态
- 当前仓位与挂单
- 当前 symbol 风险预算
- 当前 regime 标签

状态刷新建议：

- 50ms：BBO 观察缓存更新
- 100ms：短周期状态增量更新
- 200ms：盘口特征与 books5 状态更新
- 1s：高层 regime 更新

说明：

状态引擎优先保证轻量、稳定、可恢复，不在热路径中执行复杂检索、AI 调用或大规模持久化。

### 6.3 Memory / Audit / Checkpoint System

V1 统一把记忆、审计和检查点视为 Harness 的持久化层，而不是零散的辅助模块。

包括三部分：

#### 短期热记忆

- 当前状态缓存
- 当前快照版本
- 当前预算池
- 当前异常标记

建议使用：
- **Redis**

#### 长期结构化审计

- 订单
- 成交
- 仓位
- 风控事件
- 案例
- 版本
- 评测记录

建议使用：
- **PostgreSQL**

#### 相似案例检索

- 爆亏案例
- 假突破案例
- 风控否决案例
- 特定市场环境下历史经验

建议使用：
- **Qdrant**

### 6.4 Expert Layer

V1 建议先实现 5 个专家角色：

- 市场结构专家
- 事件过滤专家
- 策略路由专家
- 风险专家
- 审计专家

说明：

- 专家输出必须符合 `ExpertOpinion` 结构
- Expert Layer **不进入逐笔交易热路径**
- 它只负责低频评估市场环境、生成配置建议、更新白名单/黑名单、调整风险倍数与策略开关

其运行节拍不应假设为秒级稳定响应。更合理的 V1 设计是：

- **常规周期：30s ~ 5min**
- **事件触发：重大异常、策略切换、风控状态变化时额外触发一次**

### 6.5 Signal Factory

V1 先做 3 个基础策略：

#### S1 微突破策略

适用于：

- 波动压缩后扩张
- BBO 连续抬升/压低
- 主动成交一侧强化

#### S2 短线回归策略

适用于：

- 盘口短时失衡后回归
- 超短期超买超卖
- 波动未明显扩张

#### S3 异常风险暂停策略

用途：

- 非盈利型保护策略
- 发现盘口异常、成交异常、回报异常时暂停动作

#### 止损机制

V1 统一采用：

- **ATR 止损 + 最大百分比止损双保险**

执行原则：

- 优先根据 ATR 设定动态止损距离
- 同时设置最大百分比止损上限，防止 ATR 在异常环境下过宽
- 若两者冲突，取更保守的一侧执行
- 止损逻辑由确定性规则执行，不交由 AI 临场决定

统一信号格式：

- `symbol`
- `strategy_id`
- `side`
- `entry_type`
- `urgency`
- `confidence`
- `max_hold_ms`
- `cancel_after_ms`
- `risk_tag`

### 6.6 Decision Committee

职责：

1. 整合专家意见
2. 发布新的 `StrategyConfigSnapshot`
3. 管理当前交易模式
4. 决定某些策略是否启用
5. 决定某个 symbol 是否暂停/恢复

它不是逐笔审批引擎，而是：

**低频配置发布器**

逐笔交易时，Execution Engine 不等待 Decision Committee 即时裁决，而是直接使用最近一版已生效的配置快照。

### 6.7 Execution Engine

职责：

- 下单
- 撤单
- 改单
- 状态同步
- 超时撤单
- 断线恢复
- 订单去重与重试
- 检查点恢复后的对账与补齐

关键原则：

#### 原则 1：执行完全确定性

执行层不可由 LLM 自由生成动作。

#### 原则 2：显式动作预算池

每个 instrument 维护：

- place budget
- cancel budget
- amend budget

V1 默认应故意低于官方理论限频运行。

#### 原则 3：以 taker 为主，maker 作为后续扩展

V1 执行风格明确为：

- 以 taker 成交确定性为优先
- maker 不作为 V1 主执行模式
- 不在 V1 中追求频繁挂撤优化
- 如未来验证稳定后，再评估少量 maker 场景

#### 原则 4：网络抖动优先纳入执行设计

执行层必须显式处理网络与交易所抖动，而不是假设连接持续稳定。V1 应支持：

- WebSocket 心跳检测与超时重连
- 私有与公共连接分离
- 本地状态快照与恢复
- 下单幂等键与 client order id 去重
- 短暂断线后的状态补齐
- 网络异常时切换到保护模式

建议加入以下降级模式：

- `normal`：正常交易
- `degraded`：只允许低频开仓与正常平仓
- `reduce_only`：只允许减仓，不允许新增风险
- `halted`：冻结全部新动作，等待人工或系统恢复

建议复用：

- **NautilusTrader** 作为执行事件框架参考或底座
- OKX 官方交易接口作为最终执行通道
- **CCXT** 可用于研究工具与离线脚本，不建议作为高频执行核心

### 6.8 Risk Kernel

这是最重要的硬约束层，独立于 AI。

V1 建议硬编码以下规则：

- 单笔最大风险 <= 0.35% NAV
- 单 symbol 最大敞口 <= 12% NAV
- 总敞口 <= 30% NAV
- 连续 3 笔亏损暂停该 symbol
- 日内亏损 >= 2.0% 切换只减仓
- 连续接口异常 >= 5 次冻结新开仓
- 行情异常 / 数据不同步 / 回报异常时触发熔断

新增稳定性风控：

- 连续 WebSocket 重连失败时自动进入 `reduce_only`
- 本地状态与交易所状态不一致时禁止新开仓
- AI 层超时或不可用时自动切换为“纯规则模式”
- 通知通道异常不影响主交易链路，但必须记录高优先级告警待补发

---

## 7. 节拍设计

V1 节拍采用“稳定优先、可达优先”版本，只保留工程上长期稳定可实现的节拍，并主动为网络波动、交易所抖动、AI 调用波动和异常恢复留出冗余。

### 7.1 保留的节拍目标

#### 执行与状态热路径层

- 行情观察：50ms ~ 200ms
- 状态更新：100ms ~ 200ms
- 订单回报：事件驱动

#### 快路径规则层

- 规则策略判定：250ms ~ 500ms
- 风控硬闸：逐笔同步执行

#### 慢路径治理层

- AI 治理节拍：**30s ~ 5min，或事件触发**
- 事件触发场景：
  - 风控模式切换
  - 策略切换
  - 交易所连接异常
  - 状态一致性异常
  - BTC / ETH 异常波动
  - 预定义宏观事件窗口

#### 执行层撤单管理

- 事件驱动
- 超时撤单可配置在 500ms ~ 1500ms

### 7.2 主动移除的目标

V1 明确移除以下不适合当前阶段的设计：

- 每 10ms 进行一次完整专家投票
- 每 10ms 进行一次 LLM 参与的交易决策
- 每 10ms 进行数据库重写、案例检索或记忆召回
- 以免费 OKX 接口追求极限 HFT 级别的持续挂撤循环
- 将 AI 放入逐笔执行、撤单、改单的热路径
- 假设 GPT 可以稳定支持秒级治理闭环

### 7.3 V1 的现实频率定位

因此，V1 的正确定位应定义为：

- 行情观察：50ms ~ 200ms
- 状态更新：100ms ~ 200ms
- 规则决策：250ms ~ 500ms
- 执行响应：事件驱动 + 500ms ~ 1500ms 撤单管理
- AI 治理层：30s ~ 5min，或事件触发

这意味着 V1 面向的是：

**稳定优先的自动化短线 + 中高频执行 + 低频 AI 治理层**

而不是：

**LLM 驱动的极限高频交易系统**

## 8. 稳定性、通知与恢复设计

### 8.1 网络波动处理

V1 必须把网络波动视为常态而不是异常，建议加入：

- WebSocket 心跳检测、读超时、写超时与自动重连
- 公共行情流与私有交易流分离
- 重连后的状态补齐与挂单校验
- `ExecutionCheckpoint` 恢复
- 网络质量指标采集：延迟、抖动、丢包、重连次数
- 网络异常触发降级模式，而不是继续正常开新仓

### 8.2 AI 调用波动处理

AI 调用存在天然的延迟波动与偶发失败，因此必须与主交易链解耦：

- AI 只参与高层路由、风险调权、审计与复盘
- V1 采用标准版 AI 参与深度：
  - 策略路由
  - 风险调权
  - 审计与复盘
  - 记忆检索
  - 候选优化方案生成
- AI 不参与逐笔热路径
- 设定更宽松的固定超时，例如 **8s ~ 20s** 的调用预算
- 超时后直接丢弃本次 AI 输出，不阻塞规则层
- AI 调用失败时自动退回 `rules_only_mode`
- AI 输出必须带时间戳和有效期，过期结果不可复用
- AI 调用结果写入异步队列，不直接写入热路径数据库

建议增加两个运行模式：

- `hybrid_mode`：规则层 + AI 治理层协同
- `rules_only_mode`：AI 不可用时自动切换，仅保留规则与风控

### 8.3 通知与告警渠道

系统需要把开仓、平仓、风险异常与关键市场状态主动通知给用户。V1 明确采用 **Telegram Bot** 作为统一通知与交互主通道。

#### 一级：即时通知

适合发送：

- 开仓
- 平仓
- 止损触发
- 熔断触发
- 切换到 `reduce_only` 或 `halted`
- AI 不可用并切换到 `rules_only_mode`
- 交易所连接异常或状态不一致

#### 二级：关键市场状态与重要事件通知

适合发送：

- 市场波动显著放大
- 当前 symbol 暂停交易或恢复交易
- 策略切换
- 进入 `degraded` / `reduce_only` / `halted`
- funding / OI / mark 偏离达到预设阈值
- 预定义重要事件窗口触发

V1 中“重要事件”明确限定为：

- funding 异常
- OI 异常
- mark 偏离异常
- 交易所连接异常
- 策略切换
- 进入或退出 `reduce_only` / `halted`
- 预定义宏观时间窗口
- BTC / ETH 的重大异常波动

#### 三级：查询型通知

适合让用户主动查看：

- 当前仓位
- 最近 10 笔交易
- 今日收益摘要
- 当前系统模式
- 当前 AI / 网络 / 交易所健康状态

建议预留以下 Bot 命令：

- `/status`
- `/positions`
- `/orders`
- `/risk`
- `/pnl`
- `/mode`
- `/market`

### 8.4 通知分级建议

建议通知至少分为：

- `INFO`：普通开平仓、状态切换
- `WARN`：网络抖动增加、AI 超时增多、单 symbol 暂停
- `CRITICAL`：熔断、只减仓、状态不一致、接口异常累积

注意：

- 通知系统必须异步，不得阻塞主交易链路
- 通知失败只记录与补发，不影响下单与风控执行
- CRITICAL 级事件应支持 Telegram Bot 重试与补发

## 12. 策略研究与 Regime 路由框架

### 12.1 策略设计原则

V1 不追求“单一利润最大化策略”，而是追求：

- **风险调整后收益最大化**
- **成本后仍有正期望**
- **在不同市场阶段具备存活能力**
- **可解释、可回放、可晋升、可回滚**

因此，V1 采用 **策略篮子 + Regime 路由** 的设计，而不是押注单一“圣杯策略”。

### 12.2 V1 策略篮子

V1 默认采用三类基础策略：

#### S1 趋势 / 突破策略
适用于：
- 趋势一致性较强
- 波动由压缩转向扩张
- 主动成交一侧持续增强
- 市场未进入明显过热反转状态

核心目标：
- 捕捉短线趋势延续与突破扩张

#### S2 均值回归策略
适用于：
- 市场缺乏明确方向趋势
- 盘口短时失衡后回归概率较高
- 波动未明显扩张
- 偏离主要来自短期流动性扰动而非新信息

核心目标：
- 捕捉短时偏离后的价格回归

#### S3 异常暂停 / 禁入策略
用途：
- 不是盈利策略，而是保护策略
- 当市场、交易所或系统处于异常状态时，主动暂停某些方向或全部新增风险

适用场景：
- 盘口跳变异常
- funding / OI / mark 偏离过强
- 交易所连接异常
- 状态一致性异常
- 宏观事件窗口

### 12.3 为什么采用策略篮子

因为 BTC / ETH 的短线市场不会长期只适合一种策略：

- 趋势阶段中，突破策略通常更优
- 横盘震荡阶段中，回归策略通常更优
- 异常阶段中，最优策略往往是“不交易”

因此系统的关键能力不是“预测一切”，而是：

**识别当前更像哪一类市场状态，并路由到更合适的策略。**

### 12.4 Regime 路由器

Regime 路由器是慢路径治理层中的核心组件，负责回答：

- 当前市场更偏趋势还是震荡
- 当前波动是压缩、扩张还是异常
- 当前 funding / OI / mark 状态是否支持继续交易
- 当前是否应该降低风险、暂停某个 symbol，或切换到保护模式

V1 的 Regime 特征建议先从以下几类构建：

#### 趋势特征
- 短中周期收益方向一致性
- 突破后的延续性
- 趋势持续时间

#### 波动特征
- 波动压缩 / 扩张
- 异常跳变
- 波动变化速率

#### 微观结构特征
- spread
- order book imbalance
- 主动成交偏向
- 短时流动性失衡

#### 衍生品特征
- funding 状态
- OI 变化
- mark 偏离状态

#### 系统特征
- 网络抖动
- 订单回报延迟
- 状态一致性健康度
- 当前是否处于降级模式

### 12.5 Regime 到策略的路由逻辑

V1 建议先采用简单、可解释的路由：

- **趋势明显 + 波动扩张 + 非异常** -> 优先启用 `S1 趋势/突破`
- **无明显趋势 + 失衡可回归 + 非异常** -> 优先启用 `S2 均值回归`
- **异常波动 / 衍生品过热 / 系统异常** -> 启用 `S3 异常暂停/禁入`

这层逻辑不追求花哨，而追求：

- 可解释
- 可回放
- 可评测
- 可逐步替换为更复杂模型

### 12.6 策略研究方法论

V1 中寻找可盈利策略，不采用“先让 AI 猜一个利润最大策略”的方式，而采用：

1. **提出可解释策略假设**
2. **定义结构化特征与适用条件**
3. **历史回放与仿真验证**
4. **做成本 / 滑点 / 延迟压力测试**
5. **分市场阶段评估表现**
6. **进入 paper trading**
7. **进入小资金实盘**
8. **做归因、晋升与回滚**

换句话说，V1 要找的不是“回测最赚钱的策略”，而是：

**成本后仍有正期望、在多个阶段不过度失真、并且能够稳定执行的策略。**

### 12.7 AI 在策略研究中的角色

AI 在策略层不负责逐笔择时，而更适合负责：

- 整理和解释市场状态
- 归纳当前 regime 的证据
- 生成策略配置建议
- 复盘某类策略为何失效
- 生成 challenger 候选研究方向

不适合负责：

- 在 250ms ~ 500ms 窗口内决定下一笔交易
- 决定实时撤单/改单是否立即发生
- 直接替代风控内核

### 12.10 V1 策略层结论

V1 的策略设计不追求“找到唯一最赚钱策略”，而是追求：

- 建立一个可切换的策略篮子
- 建立一个可解释的 Regime 路由器
- 在不同市场状态下启用更合适的策略
- 用治理层和风控层限制策略误用

因此，V1 的策略系统正确定位应为：

**策略篮子 + Regime 路由 + 风控治理，而不是单一策略押注。**

## 13. Harness 评测与自我迭代机制

系统采用“交易循环 + 学习循环 + 晋升循环”三段结构。

### 12.1 交易循环

职责：

- 实时运行策略
- 下单与风控
- 记录所有决策与结果
- 写入检查点与审计记录

### 12.2 学习循环

职责：

- 汇总当日交易日志
- 生成失败模式
- 产出候选参数与候选提示词
- 召回相似历史案例
- 更新专家评分
- 生成新 challenger 版本

注意：

V1 中学习循环只能生成候选版本，不能直接修改生产系统。

### 12.3 晋升循环

版本状态：

- `draft`
- `offline_passed`
- `sim_passed`
- `paper_passed`
- `canary_live`
- `production`
- `rollback`

晋升原则：

- 新版本必须优于当前 champion 才能进入下一阶段
- 任意阶段出现异常风险，可自动回滚

### 12.4 Harness 评测重点

V1 评测重点不是只评估“模型说得对不对”，而是评估整个 Harness：

- 配置快照是否稳定
- 风险调权是否降低回撤
- 事件过滤是否减少误交易
- 降级模式是否及时触发
- 恢复流程是否能正确对账
- AI 治理建议是否带来净正贡献

## 15. 部署架构与运维方案

部署问题在本系统中非常关键，因为它直接影响：

- 网络稳定性
- 执行延迟
- 数据库可靠性
- 恢复能力
- 运维复杂度
- 成本控制

V1 不建议一开始做过度复杂的多地域高可用集群，而建议采用：

**单主交易节点 + 独立数据/监控组件 + 明确的恢复与回滚机制**

### 15.1 V1 推荐部署形态

V1 推荐采用 **云服务器 + 容器化部署** 的方案，优先保证：

- 稳定
- 易维护
- 易重启
- 易备份
- 易迁移

推荐形态：

#### 方案 A：单云主机 + Docker Compose（V1 推荐）
适合：
- 早期原型
- 模拟盘
- 小资金实盘
- 单人或小团队维护

部署结构：
- `trader`：主交易进程
- `auditor`：审计/复盘进程
- `redis`：热状态缓存
- `postgres`：结构化数据存储
- `qdrant`：案例检索
- `prometheus`：指标采集
- `grafana`：监控面板

优点：
- 结构简单
- 容易落地
- 迁移成本低
- 回滚方便

缺点：
- 单机故障风险较高
- 不适合更大规模扩展

#### 方案 B：单主交易节点 + 托管数据库（V1.5 可考虑）
适合：
- 小资金实盘进入稳定期
- 想降低数据库运维成本

部署结构：
- 交易核心服务仍部署在单独云主机
- PostgreSQL / Redis 可迁移到托管服务
- 监控和可视化可独立部署

优点：
- 数据层更稳
- 备份、恢复更方便
- 降低数据库维护负担

缺点：
- 成本上升
- 架构更复杂

#### 方案 C：Kubernetes / 多节点集群（V2 以后再考虑）
V1 不建议直接上 Kubernetes，原因：
- 复杂度高
- 故障面更多
- 会把大量精力消耗在平台运维上
- 对当前阶段收益不明显

### 15.2 部署位置建议

V1 的部署地点选择原则：

- 优先选择网络访问 OKX 稳定、延迟较低的地区
- 优先保证连接质量，而不是追求极端低延迟
- 优先选择你自己维护起来更顺手的云平台

建议原则：
- **主交易节点和 Redis 尽量同机或同可用区**
- **交易节点与 PostgreSQL / Qdrant 尽量低延迟连接**
- **Telegram Bot 与监控组件可以和主交易节点同区部署**

V1 不建议：
- 多地域部署主交易节点同时活跃交易
- 跨洲数据库访问
- 把热路径服务拆到太多机器上

### 15.3 服务拆分建议

V1 推荐至少拆成以下服务：

#### 1）Trader Service
职责：
- Market Bus
- State Engine
- Signal Factory
- Risk Kernel
- Execution Engine

说明：
- 这是热路径核心服务
- 优先保证独立、轻量、少依赖

#### 2）Governor Service
职责：
- Expert Layer
- Decision Committee
- StrategyConfigSnapshot 生成
- 低频 AI 调用

说明：
- 这是慢路径治理服务
- 可以单独重启，不应影响快路径已运行配置

#### 3）Auditor Service
职责：
- 审计日志整理
- 案例沉淀
- 复盘生成
- HarnessEvalRecord 生成

#### 4）Notifier Service
职责：
- Telegram Bot 推送
- Telegram 命令查询
- 消息补发与节流

#### 5）Replay / Backtest Service
职责：
- 历史回放
- 仿真验证
- challenger 测试

说明：
- 不进入生产热路径
- 可按需启动

### 15.5 上线与回滚方式

V1 建议采用：

- **Git + 容器镜像版本化**
- **配置文件版本化**
- **StrategyConfigSnapshot 版本化**
- **一键回滚到上一版镜像和配置**

上线流程建议：
1. 代码合并到主分支
2. 构建镜像
3. 在仿真/模拟环境验证
4. 发布到生产节点
5. 先启用新版本慢路径或只读模式
6. 确认状态一致后再恢复正常模式

回滚原则：
- 代码回滚
- 配置回滚
- 快照版本回滚
- 必要时直接进入 `reduce_only` 或 `halted`

### 15.6 检查点与恢复流程

恢复流程建议：
1. 读取最近 `ExecutionCheckpoint`
2. 恢复当前模式、仓位快照、预算池快照
3. 重新连接 OKX 公共/私有流
4. 拉取当前未完成订单与实际持仓
5. 与本地状态做对账
6. 若一致则恢复交易
7. 若不一致则进入 `reduce_only` 或 `halted`

说明：
- 恢复优先于继续交易
- 对账成功前不得新增风险

### 15.7 监控与运维指标

建议至少监控：

#### 系统指标
- CPU
- 内存
- 磁盘
- 容器重启次数

#### 网络指标
- WebSocket 重连次数
- 延迟
- 抖动
- 接口错误率

#### 交易指标
- 下单成功率
- 撤单成功率
- 状态不一致次数
- 风控触发次数
- `reduce_only` / `halted` 触发次数

#### AI 治理指标
- AI 调用超时率
- 慢路径发布快照频率
- 快照过期次数
- `rules_only_mode` 切换次数

### 15.8 V1 部署建议结论

V1 最推荐的落地方案是：

**一台主交易云服务器 + Docker Compose + Redis/PostgreSQL/Qdrant + Telegram Bot + Prometheus/Grafana**

如果你想进一步稳一点，可以演进到：

**主交易节点单独部署 + PostgreSQL/Redis 托管化 + Auditor/Replay 独立节点**

V1 不建议直接上复杂集群。

## 16. 开发阶段规划

### 阶段 1：Harness 基础闭环

目标：

- 打通行情、状态、信号、下单、风控、回报、审计
- 支持事件回放与模拟盘
- 接入 Telegram Bot 基础通知
- 实现 `StrategyConfigSnapshot` 与 `ExecutionCheckpoint`
- 先验证稳定可跑与状态一致性

### 阶段 2：慢路径治理层

目标：

- 接入 OpenAI Agents SDK
- 建立案例库与检索系统
- 建立专家评分体系
- 将 AI 接入策略路由、风险调权、审计与复盘

### 阶段 3：Harness Eval 与晋升机制

目标：

- 接入 MLflow
- 建立 challenger 流程
- 建立 harness 级评测指标
- 自动生成候选优化方案

### 阶段 4：小资金实盘

目标：

- 仅上线 BTC / ETH
- 先上少量资金
- 先上单向仓位、taker 主执行
- 以风控稳定性、通知可靠性与系统健壮性为优先指标

#### 阶段晋升门槛

从模拟盘进入下一阶段前，至少满足：

- 模拟盘连续运行 7 天无严重系统异常
- 订单状态一致性校验通过
- 风控触发逻辑与降级模式验证通过
- Telegram Bot 的关键通知无漏报
- 审计与回放链路完整可用
- 检查点恢复流程通过演练

## 17. 当前版本结论

本方案的优势在于：

- 贴合 OKX 免费接口限制，现实可落地
- 最大化复用成熟开源免费组件，降低开发成本
- 把高风险部分锁定在确定性执行与硬风控内
- 把 AI 放到更合理的位置：低频治理、审计、记忆与进化
- 具备 Harness 所需的关键能力：快慢路径、契约、快照、检查点、降级、评测与晋升

本方案的边界在于：

- V1 不追求顶级 HFT
- V1 不做自动改代码并直接上线
- V1 不做大规模全网新闻理解
- V1 不做激进强化学习实盘控仓
- V1 不假设 GPT 可以稳定支持秒级在线治理

因此，V1 的正确目标应定义为：

**构建一个以 BTC / ETH 的 OKX 永续合约短线为主、以 taker 为主执行、单向持仓管理、具备快慢路径分离、配置快照、执行检查点、低频 AI 治理与持续进化能力的 Trading Harness 底座。**
